# Поиск релевантных статей справки для RAG-пайплайна поддержки Авито

## Общий подход

Гибридный поиск: лексический (BM25) и семантический (dense эмбеддинги) каналы объединяются в единый ранжированный список, который затем дополнительно уточняется обучаемой моделью ранжирования (learned reranker), обученной на 500 размеченных примерах из calibration.f.

Готовые cross-encoder реранкеры ухудшали метрику, тогда как специально настроенные под конкретный домен и язык компоненты (BM25 с русской лемматизацией, мультиязычные/русскоязычные эмбеддеры, обучаемая модель ранжирования на
собственных признаках) стабильно её улучшали. Финальное решение — это результат нескольких итераций проб и отказа от подходов, которые интуитивно должны были помочь, но не помогли на практике.

## Как запустить

1. Данные

Положить articles.f, calibration.f, test.f в папку candidate_data/
в корне проекта (путь задан в config.py, DATA_DIR).

2. Установка зависимостей
```
python -m venv .venv && source .venv/bin/activate
pip install torch --index-url https://download.pytorch.org/whl/cpu
pip install -r requirements.txt
```

3. Построение индексов

`python -m src.build_indexes`

Строит BM25-индекс и два dense FAISS-индекса (по телу и по заголовку статьи) в папке indexes/. Нужно перезапускать при любом изменении articles.f, модели эмбеддингов или связанных гиперпараметров (TITLE_REPEAT, EMBEDDING_MODEL_NAME, EMBEDDING_MAX_SEQ_LENGTH).

4. Обучение learned reranker

`python -m src.train_reranker`

Обучает LightGBM LambdaMART на признаках из calibration.f, сохраняет модель в indexes/learned_reranker.pkl. В выводе печатается out-of-fold MAP@10 по 5 фолдам.

5. Проверка качества на калибровке

```
python -m src.evaluate  # итоговая MAP@10
python -m src.evaluate --recall  # recall@K по BM25/body-dense/title-dense/RRF
python -m src.evaluate --tune  # перебор гиперпараметров fusion
python -m src.evaluate --tune --skip-reranker  # то же самое, но без загрузки cross-encoder
```

Этот шаг не обязателен для получения answer.csv.

6. Генерация ответа на test.f

`python -m src.predict`

Прогоняет test.f через тот же HybridRetriever с гиперпараметрами из config.py, валидирует результат и сохраняет answer.csv в корне проекта.

## Разведочный анализ данных (EDA)

Корпус статей - 793 записи (article_id, title, body), калибровка и тест - по 500 запросов каждая. Заголовки короткие и однородные: средняя длина 29 символов, максимум 113. Тела статей после очистки от HTML сильно разнородны по длине: средняя длина 5136 символов, медиана 2240, но максимум - 506100 символов.
Сортировка статей по длине показала, что помимо этого экстремального выброса (статья со списком популярных категорий товаров) есть ещё десяток статей длиной 25-80 тысяч символов, которые при ручной проверке оказались содержательными многотемными материалами (например, "Как продавать и покупать с доставкой", "Защита сделки"), их решено было оставить в индексе, тогда как единственный явный выброс исключён.

Распределение числа правильных статей на один запрос в калибровке показало, что большинство вопросов имеют один правильный ответ, но заметная доля - два и более, что подтверждает важность именно ранжирования, а не только попадания в топ-1.

Анализ частотности лемматизированных токенов в запросах и в статьях показал ожидаемое доменное ядро лексики (доставка, заказ, объявление, деньги, товар, оплата, отправить, вернуть), это подтвердило, что BM25 по леммам должен быть работоспособным базовым каналом поиска для этого корпуса. Отдельно был построен рейтинг самых часто встречающихся в ground_truth статей: несколько статей (в первую очередь про доставку и оплату) фигурируют как правильный ответ существенно чаще остальных - то есть распределение целевых статей несбалансированно, что означает, что улучшение ранжирования именно для этой небольшой группы часто встречающихся статей даёт непропорционально больший вклад в итоговую метрику, чем работа с редкими статьями.

## Обработка HTML статей

Статьи в articles.f хранятся с HTML-разметкой: таблицы, вкладки, спойлеры, служебные теги. Для очистки использовался BeautifulSoup с парсером lxml: из тела статьи удалялись теги script/style/noscript,после чего извлекался текст через get_text с разделителем-пробелом и схлопыванием повторяющихся пробелов в один.

Для BM25-индекса заголовок и тело статьи объединялись в единый текст, при этом заголовок дублировался несколько раз (TITLE_REPEAT). Это эвристика для усиления его веса в лексическом поиске, поскольку заголовок обычно самый информативный сигнал о теме статьи.

Для dense-индексов заголовок и тело обрабатывались раздельно: строился отдельный текст только из заголовка и отдельный только из очищенного тела статьи, без объединения. Это привело к появлению двух независимых dense-индексов вместо одного, что улучшило метрику на calibration.f.

В ходе анализа ошибок обнаружилась одна статья-аномалия: 2924 со служебным содержанием (перечень категорий товаров), длина текста которой превышала медианную длину статьи более чем в 200 раз. Такой объём текста превращал её dense-эмбеддинг в бессмысленный средний вектор и искажал нормализацию длины в BM25. Эта статья была явно исключена из индекса списком EXCLUDED_ARTICLE_IDS в конфигурации, остальные длинные статьи были сохранены в индексе после ручной проверки: они содержали релевантный контент, просто были многотемными по своей природе.

## Признаки, модели и алгоритмы

### Лексический поиск

BM25Okapi (библиотека rank_bm25) поверх лемматизированных токенов. Токенизация - простой regex по кириллице/латинице/цифрам, лемматизация - через pymorphy2. Лемматизация была необходима, поскольку без неё BM25 не сопоставлял бы разные словоформы одного слова друг с другом.

### Семантический поиск

Использовались два независимых dense-индекса на базе FAISS (IndexFlatIP, косинусная близость через нормированные векторы):

- Body-индекс - эмбеддинг тела статьи, основной канал, участвующий в объединении с BM25.
- Title-индекс - эмбеддинг только заголовка статьи, короткий и точный   сигнал, использующийся исключительно как дополнительный признак для обучаемого реранкера, не участвующий в основном ранжировании кандидатов.

Модель эмбеддингов менялась в ходе экспериментов. Первая версия (intfloat/multilingual-e5-base, мультиязычная, ~278M параметров) дала recall заметно ниже, чем у BM25. Замена на deepvk/USER-base, а затем на специализированную под русский язык модель deepvk/USER-bge-m3, увеличила recall dense-канала и подняла итоговую
метрику. Отдельно потребовалось ограничить максимальную длину последовательности модели: BGE-M3 по умолчанию поддерживает контекст до 8192 токенов, что на CPU делало инференс на порядок медленнее без заметного прироста качества; ограничение до 1024 токенов вернуло приемлемую скорость. Для этого также была проведена проверка, что качество предсказания и метрика AP@10 не зависит от длины статьи, т.е. длинные статьи не предсказываются хуже  коротких.  

### Объединение каналов (fusion)

Проверялось два способа объединения ранжированных списков BM25 и dense:

- Reciprocal Rank Fusion (RRF) - учитывает только позицию документа в   каждом списке, не абсолютную величину скора.
- Score-based fusion - нормализация (min-max) скоров каждого метода в пределах одного запроса и взвешенное суммирование.

Score-based fusion на калибровке стабильно превзошёл RRF: абсолютная величина скора несёт полезный сигнал, который чисто ранговый RRF терял. Вес BM25 относительно dense канала подбирался на калибровке; неожиданный результат экспериментов - равный вес (1.0/1.0) оказался лучше искусственного увеличения веса BM25, несмотря на то что BM25 сам по себе показывал более высокий recall - вероятно, из-за формы распределения нормализованных скоров BM25 после min-max, которая и так неявно отдавала ему приоритет.

### Реранкинг

Было испробовано три принципиально разных подхода к дополнительному ранжированию top-N кандидатов после fusion:

1. **Готовый англоязычный cross-encoder (cross-encoder/ms-marco-MiniLM-L6-v2)** резко ухудшил метрику. Причина выяснилась быстро: модель обучена исключительно на английском MS MARCO и не понимает русский текст, её скоры оказались фактически случайными.

2. **Готовый мультиязычный cross-encoder (cross-encoder/mmarco-mMiniLMv2-L12-H384-v1)**, обученный в том числе на русских данных mMARCO. Деградация метрики сохранилась, хотя и менее выраженная. Модель обучена ранжировать абзацы веб-страниц по поисковому запросу, а не статьи техподдержки по разговорному вопросу пользователя. Жанровое несовпадение оказалось существеннее, чем ожидалось, и добавление такого реранкера снижало MAP@10 тем сильнее, чем больше кандидатов ему передавалось на пересортировку.

3. **Обучаемая модель ранжирования (learned reranker)** на признаках, построенных из уже посчитанных сигналов: сырые и нормализованные скоры BM25/body-dense/title-dense, обратные ранги в каждом списке, лексическое пересечение запроса с заголовком статьи, флаг присутствия кандидата одновременно в обоих базовых списках, длина запроса в токенах. Первая версия на LogisticRegression с точечной (пара запрос-документ, бинарная метка) целью показала высокий ROC-AUC (~0.82), но не улучшила MAP@10. Логичное расхождение между точечной классификацией и ранжированием внутри запроса. Замена на LightGBM с объективом lambdarank (LambdaMART), который оптимизирует ранжирующую метрику напрямую и учитывает группировку кандидатов по запросу, устранила это расхождение, но потребовала отдельного решения проблемы переобучения.

Оба готовых cross-encoder реранкера в финальном решении отключены (USE_RERANKER = False); используется только обучаемый LambdaMART-реранкер (USE_LEARNED_RERANKER = True).

## Проверка качества на calibration.f

Основная метрика - MAP@10 из задания.

Кроме итоговой MAP@10 использовались дополнительные диагностики, без которых часть проблем осталась бы незамеченной:

Recall@K по отдельным источникам кандидатов (BM25, dense, их объединение) на K=10/20/30, без учёта реранка. Recall@30 был равен 0.9, Recall@20 0.85, Recall@10 - 0.72. Эта диагностика показала, что разрыв между Recall@10 и Recall@30 составлял около 15-20 процентных пунктов на протяжении всех экспериментов, то есть нужная статья почти всегда находилась среди кандидатов, но не всегда попадала в первую десятку. Это направило дальнейшие усилия именно на улучшение ранжирования, а не на расширение пула кандидатов.

Также был реализован Grid-search по гиперпараметрам fusion (метод объединения, вес BM25, параметр сглаживания RRF), чтобы подобрать наилучшие комбинации.

Для обучаемого реранкера сделано разбиение калибровки на 5 фолдов по query_id (чтобы кандидаты одного запроса не попадали одновременно в train и val), с оценкой не по AUC классификации, а по out-of-fold MAP@10: для каждого валидационного запроса модель, обученная без него, прогонялась через весь пайплайн. Это было критично, поскольку первая версия эксперимента показала разрыв: MAP@10, посчитанный на калибровке моделью, обученной на всей
калибровке (0.51), оказался сильно завышен относительно результата на тесте (0.35), т.е. была утечка данных. После введения out-of-fold оценки и раннего останова обучения (early stopping) на отдельном holdout-сплите разрыв между калибровкой и тестом сократился до уровня шума.

## Типы ошибок, найденные при анализе, и что с ними сделано

Для системного разбора ошибок каждое предсказание на калибровке было классифицировано по типу несоответствия ground truth в топ-10:

Perfect — все правильные статьи найдены и корректно расположены.
Ranking — хотя бы одна правильная статья присутствует в топ-10, но первая позиция предсказания неправильная.
Partial — первая позиция предсказания верна, но не все правильные статьи из ground truth найдены (актуально для многодокументных запросов).
Miss — ни одной правильной статьи нет в топ-10.

Распределение по 500 запросам калибровки: Ranking — 230 запросов (46%), Miss — 102 (20%), Perfect — 97 (19%), Partial — 71 (14%). Ranking -самая массовая категория ошибок, что количественно подтверждает наблюдение из recall-диагностики: система в целом находит нужные статьи, но систематически неверно расставляет их порядок внутри топ-10. Это стало основным аргументом в пользу того, чтобы вкладывать усилия в улучшение ранжирования, а не только в расширение пула кандидатов.

Отдельно был проанализирован более широкий список кандидатов (не обрезанный до топ-10, а включающий более глубокие позиции), чтобы проверить, есть ли среди запросов категории Miss такие, где правильная статья отсутствует вообще. Ни один из потерянных запросов не оказался полным провалом поиска: для каждого запроса из категории Miss правильная статья находилась где-то в более широком списке кандидатов, просто ниже границы топ-10. Иными словами, все зафиксированные ошибки Miss на срезе топ-10 — это ошибки глубины ранжирования, а не ошибки полного отсутствия сигнала в retrieval-этапе.

Распределение позиции первого правильного попадания показало, что для 168 из 500 запросов правильный ответ уже стоит на первом месте, но для заметной части запросов (порядка 80) первое правильное попадание приходится на позиции 6-10 — то есть решение находит документ, но затем, по метрике MAP@10, эти случаи вносят лишь небольшой вклад из-за низкой позиции.

Анализ AP@10 по конкретным article_id как целевой статье (не по запросам) выявил группу статей, которые система находит систематически хуже среднего: их отличает то, что заголовки короткие и обобщённые ("Ошибки сайта или приложения", "Что такое тариф", "Предложить услугу", "Меня обманули"), а в корпусе присутствует несколько разных статей с близкими по смыслу или почти дублирующимися заголовками на схожую тему (например, две статьи с заголовком "Ошибки сайта или приложения" под разными article_id, статьи про разные виды ошибок при подаче
объявления). Для таких групп признаки, основанные на лексическом и семантическом сходстве целиком статьи, недостаточно различают кандидатов между собой — модель путает семантически близкие, но по существу разные статьи.

Проверка зависимости AP@10 от средней длины ground truth статей (диаграмма рассеяния со скользящим средним) не показала выраженной зависимости: сглаженная кривая колеблется в диапазоне примерно 0.25-0.5 без явного тренда на всём диапазоне длин, за исключением плавной просадки в области статей длиной 15-20 тысяч символов, которая скорее
всего объясняется малым количеством наблюдений в этом диапазоне, чем системным эффектом. 

### Объяснение результата MAP@10 = 0.38 

1. Мало данных для обучения реранкера. 500 запросов - недостаточно для этой задачи: даже LightGBM с регуляризацией дал прирост очень маленький относительно простого score-fusion.
2. В корпусе есть почти дублирующиеся статьи (близкие по смыслу заголовки под разными article_id).
3. 46% ошибок Ranking, а не Miss: нужная статья почти всегда есть среди кандидатов, но конкурирует с очень похожими статьями из-за коротких, разговорных, неконкретных вопросов пользователей.
